Specification procedure


Our goal is to have complete specifications for each module we release. To achieve this goal you will need to create and completely populate a spec.xml file that lives next to the .cc file associated with a module.
 

spec.xml
Combined with the documentation extracted via Cocoon, spec.xml defines a complete specification for a module. It contains information the compile needs, as well as various levels of documentation targeted at different user levels. spec.xml Documentation will be extracted from spec.xml, it is intended to be maintained just as the C++ code is.

User levels

user
User who works with a predefined network map of modules. This level is not relevant for module level specifications. They will have a unique user guide for relevant to the network they use.
builder
User will do the above as well as building and editing their own maps, but will write no code.
developer
User will write modules they needs as well as all of the above.

Steps:

  1. Start with .cc file.  Make sure it conforms to the coding standard.
  2. Update/enhance cocoon style comments.
  3. Take a screen shot of the gui (if it has one) and convert to gif format. This should be placed in ./doc relative to the .cc file for the module.  Name it module_nameGui.gif. use xv for generating grabbing the image, it will grab the whole dialog, and just the dialog.  If you need more than one window in the grab, and you are on an SGI use snapshot, then xv to convert to gif format.
  4. Write spec.xml.  It should live in the same dir as the .cc file. See the annotated spec.xml to understand what each tag is for.
  5. The current plan is to gmake docs: generate Specification in the various formats (xml, html, latex) This is not currently implemented, so do not do this step.
